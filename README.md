# Solfeggio — Arpeggio Deployment Toolkit

Solfeggio orchestrates a full Docker-based deployment of the **Arpeggio** platform. It clones application source code, generates infrastructure cluster configurations, wires environment variables and TLS certificates, builds container images, and brings the stack online behind an Nginx gateway.

The Arpeggio platform consists of three packages:

| Package | Role |
|---------|------|
| **Armonia** | Shared TypeScript types, schema definitions, and Zod validators |
| **Maestro** | API server — persistence, business logic, HTTP routes, Kafka consumers, cron jobs, WebSocket |
| **Sinfonia** | Client SPA — admin panel, view engine, module contribution system |

Solfeggio does not contain application logic. It prepares a `deploy/` workspace with the selected modules and infrastructure, then builds and runs everything via Docker Compose.

## Prerequisites

- **Docker** with Compose v2 (`docker compose`) or standalone `docker-compose`
- **Git** — used to clone Armonia, Maestro, Sinfonia, and module repositories
- **Bash** — deployment scripts require a POSIX shell

## Quick start

```bash
# 1. Prepare deployment sources and infrastructure configuration
./deploy.sh

# 2. Build images and start the full stack
./systemUp.sh
```

After `systemUp.sh` completes, the public entry point is the Nginx gateway (default: `http://localhost:80`).

To tear down generated artifacts and start fresh:

```bash
./cleanDeploy.sh
```

## Deployment workflow

### 1. `./deploy.sh` — configure and prepare sources

Interactive wizard that:

1. Prompts for **modules** to deploy (comma-separated, e.g. `eCommerce,propertyManagement`)
2. Sets the shared **Docker internal network** name (default: `arpeggio_internal_network`)
3. Clones or updates **Armonia**, **Maestro**, and **Sinfonia** core repos plus per-module repos from GitHub
4. Generates `deploy/scripts/modules.manifest.json` and syncs Maestro build scripts
5. Configures optional and required **infrastructure clusters**
6. Copies and updates `apps/maestro/.env` into `deploy/maestro/.env`

**Required infrastructure** (always configured):

- **MongoDB** — sharded cluster with TLS; primary application database
- **Nginx gateway** — public entry point routing `/`, `/api/`, and `/ws/`

**Optional infrastructure** (prompted during deploy):

- **Kafka** — async emails, login history, API metrics, distributed cron, property-management notifications
- **Redis** — rate limiting, cron locking, WebSocket coordination, health/stats caching
- **ClamAV** — real malware scanning for file uploads (falls back to mock scanner when disabled)

Each cluster is generated by an interactive `generate-cluster.sh` script under `clusters/<name>/`. Generated compose files, certificates, and scripts are gitignored and recreated on each deploy.

### 2. `./systemUp.sh` — build and start

Validates the prepared `deploy/` folder, then:

1. Ensures the shared Docker network exists
2. Builds **Maestro** and **Sinfonia** Docker images from `apps/*/Dockerfile`
3. Generates `deploy/docker-compose.servers.yml` and `deploy/docker-compose.frontend.yml`
4. Starts infrastructure clusters (MongoDB → Redis → Kafka → ClamAV → Prometheus)
5. Starts Maestro services and the Sinfonia frontend
6. Starts the Nginx gateway last

**Maestro containers** (single image, different commands):

| Container | Command | Purpose |
|-----------|---------|---------|
| `maestroApi` | `npm run api` | REST API |
| `maestroKafka` | `npm run kafka` | Kafka consumers |
| `maestroWebsocket` | `npm run websocket` | WebSocket server |
| `maestroCron` | `npm run cron` | Scheduled jobs |
| `maestroAssistant` | `npm run assistant` | AI-channel responder |
| `maestroTelegram` | `npm run telegram` | Telegram bot long-poll + linking |

**Frontend container:**

| Container | Purpose |
|-----------|---------|
| `frontend` | Sinfonia SPA served by Nginx on port 80 inside the network |

### 3. `./cleanDeploy.sh` — reset

Removes the `deploy/` folder and generated cluster artifacts (compose files, certs, scripts). Preserves cluster `.env` files, `generate-cluster.sh` scripts, and `apps/maestro/.env`.

## Directory layout

```
solfeggio/
├── deploy.sh              # Interactive deployment preparation
├── systemUp.sh            # Build images and start the stack
├── cleanDeploy.sh         # Remove generated deploy and cluster artifacts
├── apps/
│   ├── maestro/
│   │   ├── Dockerfile     # Maestro image (Node 22, OpenCV, canvas, ffmpeg)
│   │   ├── .env           # Maestro env template (gitignored; edited by deploy.sh)
│   │   └── scripts/       # Build helpers synced to deploy/scripts/
│   └── sinfonia/
│       └── Dockerfile     # Multi-stage Sinfonia build → Nginx static serve
├── clusters/
│   ├── kafka/             # KRaft Kafka cluster (SASL_SSL)
│   ├── redis/             # Redis with Sentinel
│   ├── mongo/             # Sharded MongoDB with TLS
│   ├── clamv/             # ClamAV antivirus
│   ├── nginx/             # Reverse proxy / load balancer gateway
│   └── prometheus/        # Metrics (optional)
└── deploy/                # Generated at deploy time (gitignored)
    ├── armonia/
    ├── maestro/
    ├── sinfonia/
    ├── scripts/
    ├── docker-compose.servers.yml
    └── docker-compose.frontend.yml
```

## Nginx gateway routes

The generated gateway proxies public traffic to application containers on the shared Docker network:

| Path | Upstream | Purpose |
|------|----------|---------|
| `/` | `frontend` | Sinfonia SPA |
| `/assets/` | `frontend` | Static assets |
| `/api/` | `maestroApi` | Maestro REST API |
| `/api/auxiliary/media/` | `maestroApi` | Media uploads/downloads |
| `/ws/` | `maestroWebsocket` | WebSocket server |

See [clusters/nginx/README.md](clusters/nginx/README.md) for details after running `./deploy.sh`.

## Infrastructure clusters

Each cluster directory contains:

- `generate-cluster.sh` — interactive generator (preserved in git)
- `.env` / `.env.test` — configuration templates (`.env` is gitignored after first deploy)
- Generated artifacts — `docker-compose.yml`, certs, scripts (gitignored)

| Cluster | Default | Notes |
|---------|---------|-------|
| MongoDB | Required | Sharded cluster; TLS certs copied to `deploy/maestro/secrets/certificates/mongo/` |
| Nginx | Required | Public entry point; configured with upstream replica counts |
| Kafka | Optional | KRaft mode, SASL_SSL; see [clusters/kafka/README.md](clusters/kafka/README.md) |
| Redis | Optional | Master/replica with Sentinel |
| ClamAV | Optional | Mock scanner used when disabled |
| Prometheus | Optional | Started by `systemUp.sh` when generated |

## Environment configuration

Maestro environment is managed through `apps/maestro/.env`. During `./deploy.sh`, connection settings are written automatically based on cluster choices:

- `MONGODB_*` — router host, credentials, TLS cert paths
- `KAFKA_*` — brokers, SASL credentials, CA path (when Kafka is enabled)
- `REDIS_*` — root nodes, auth (when Redis is enabled)
- `CLAMAV_*` — host/port (when ClamAV is enabled)
- `CLIENT_HOST` — public URL via Nginx gateway
- `ENABLED_MODULES` — comma-separated module list matching the deploy selection

The finalized env is copied to `deploy/maestro/.env` for container runtime.

## Managing running services

**Maestro servers:**

```bash
cd deploy
docker compose -p servers -f docker-compose.servers.yml ps
docker compose -p servers -f docker-compose.servers.yml logs -f
docker compose -p servers -f docker-compose.servers.yml down
```

**Frontend:**

```bash
cd deploy
docker compose -p frontend -f docker-compose.frontend.yml ps
docker compose -p frontend -f docker-compose.frontend.yml logs -f
docker compose -p frontend -f docker-compose.frontend.yml down
```

**Infrastructure** (example for MongoDB):

```bash
cd clusters/mongo
docker compose ps
docker compose logs -f
```

## Image build context

Dockerfiles under `apps/` expect the **`deploy/` folder** as build context (not `apps/` itself). Both images require **armonia** alongside the application package:

```bash
# Maestro
docker build -f apps/maestro/Dockerfile -t arpeggio-maestro:latest deploy/

# Sinfonia
docker build -f apps/sinfonia/Dockerfile -t arpeggio-frontend:latest deploy/
```

`systemUp.sh` handles this automatically. Override image names with environment variables:

- `ARPEGGIO_MAESTRO_IMAGE` (default: `arpeggio-maestro:latest`)
- `ARPEGGIO_FRONTEND_IMAGE` (default: `arpeggio-frontend:latest`)

## Module repositories

Core repos are cloned from GitHub during `./deploy.sh`:

- `armonia_core`, `maestro_core`, `sinfonia_core`
- Per-module repos: `armonia_<module>`, `maestro_<module>`, `sinfonia_<module>`

Branch selection is prompted per core and per module. All default to `main`.

Available modules depend on what exists in the Skerd GitHub organization (e.g. `eCommerce`, `eCommerceMarketplace`, `propertyManagement`).
